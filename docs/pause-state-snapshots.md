# Pause State Snapshot Regression Suite

This document describes the snapshot-based regression testing strategy for the `pausable` module in the `credence_delegation` contract.

## Purpose

The pause state machine involves several interdependent storage keys (`PauseProposal`, `PauseApproval`, `PauseSignerCount`, etc.). Changes to the names or underlying types of these `DataKey` variants can break storage compatibility between contract versions. 

This suite captures the serialized JSON state of all pause-related storage after each major lifecycle step. Any unintentional change to the storage layout will cause these tests to fail.

## Snapshot Locations

Verified snapshots are stored in:
`contracts/credence_delegation/test_snapshots/test_pausable_state/`

## Running Tests

To run the regression suite without updating snapshots:
```bash
cargo test -p credence_delegation test_pause_proposal_lifecycle_snapshots
```

## Refresh Protocol

If you intentionally modify the storage layout (e.g., renaming a `DataKey` variant or changing a field type), you must refresh the snapshots and verify the diff.

1. **Modify Code**: Make your intended changes to `lib.rs` or `pausable.rs`.
2. **Review Diff**: Run the tests. They should fail.
   ```bash
   cargo test -p credence_delegation test_pause_proposal_lifecycle_snapshots
   ```
3. **Inspect Failures**: Carefully review the output diff to ensure that ONLY the intended changes are reflected in the storage state.
4. **Accept Changes**: If the diff is correct, update the snapshots:
   ```bash
   INSTA_UPDATE=always cargo test -p credence_delegation test_pause_proposal_lifecycle_snapshots
   ```
5. **Commit**: include the updated `.snap` files in your pull request.

## Storage Serializer

The snapshots are generated by a custom serializer in `contracts/credence_delegation/src/test_pause_snapshots.rs`. It iterates through all known pause-related keys in the contract's instance storage and captures their values in a stable, sorted JSON format using [insta](https://insta.rs/).
